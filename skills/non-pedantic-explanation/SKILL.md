# non-pedantic-explanation

어떤 개념·용어·주제를 **최대한 쉽고 정확하게** 설명하기 위한 skill.
사용자가 이 방식으로 설명해 달라고 명시적으로 요청할 때만 아래 규칙대로 답한다.

## 핵심 원칙

1. **현학적 표현 금지.** 불필요하게 어려운 단어, 전문 용어 나열, 있어 보이려는 문장을
   쓰지 않는다. 같은 뜻을 더 쉬운 단어로 쓸 수 있으면 항상 쉬운 쪽을 고른다.
2. **비유 절대 금지.** 설명하려는 대상을 전혀 다른 분야의 것에 빗대지 않는다.
   (금지 예: "함수는 요리 레시피와 같다", "메모리는 사물함이다.")
3. **직관적이고 쉬운 말.** 정의를 먼저 정확히 세우고, 그 정의만으로 이해되게 쓴다.

## 대상 독자 가정

항상 아래 사람에게 설명한다고 가정한다.

- 컴퓨터공학과 2학년 수준.
- **실무/현업에서만 쓰는 domain-specific 용어와 상용 서비스는 모른다.**
  (예: AWS의 ECR·S3·EC2, GCP·Azure의 서비스 이름 등.)
- **로컬에서 직접 해보는 기본기는 있다.**
  (예: docker container/image를 로컬에서 빌드하고 실행해 봤다.)

즉, 교과서나 toy project에서 접하는 개념은 알고, 그 위에 회사가 얹은 상용 서비스
이름은 모른다고 본다.

## 용어 사용 규칙

- 설명은 **toy project / 로컬에서 쓰는 용어**로 한다.
- 실무 용어가 꼭 필요하면, 먼저 사용자가 아는 개념으로 설명한 뒤
  "실무에서는 이걸 X라고 부른다 / X가 이 역할을 한다"로 연결한다.
- 예시: "docker hub에 온라인으로 image를 올려둘 수 있다. AWS의 ECR은 AWS 안에서
  쓰는 docker hub 같은 image 저장소다."

### 비유 금지 vs 같은 분야 대응 (헷갈리지 말 것)

- **비유(금지):** 설명 대상을 **전혀 다른 분야**의 것에 빗대는 것.
  (예: image registry를 "옷장"에 빗댐 → 금지.)
- **같은 분야 대응(허용):** 사용자가 이미 아는 것과 새것이 **같은 종류**임을 직접
  대응시키는 것. 이건 비유가 아니라 정의·분류다.
  (예: "ECR은 docker hub와 같은 종류의 image registry다" → 허용. 둘 다 실제로
  image를 저장하는 registry라서, 빗댄 게 아니라 같은 범주로 묶은 것.)

## 단계적 설명 (논리 비약 금지)

여러 논리 단계가 필요한 설명은 **아주 잘게(infinitesimally) 쪼갠다.**

- **각 단계는 바로 앞 문장에서 쓴 논리를 발판(lever)으로 삼아** 다음 단계로 넘어간다.
  앞에서 나오지 않은 사실을 다음 문장에서 갑자기 끌어오지 않는다.
- 새 사실·전제가 필요하면, 그 사실을 **먼저 한 문장으로 세우고** 그 다음 문장에서
  그걸 근거로 쓴다.
- 결론을 먼저 던지고 나중에 정당화하지 않는다. **근거 → 근거 → 결론** 순서로 쌓는다.
- 한 문장에 논리 도약을 두 개 이상 넣지 않는다. 도약이 두 개면 문장을 두 개로 나눈다.

### 예시: "docker image는 빌드가 빠르다"를 설명할 때

**나쁜 예 (비약):**
> docker image는 layer로 되어 있어서 캐시가 되니까 빌드가 빠르다.

(한 문장에 layer → 캐시 → 빠름, 도약이 세 번. 왜 layer면 캐시가 되는지, 왜 캐시가
되면 빠른지가 빠졌다.)

**좋은 예 (발판을 밟고 넘어감):**
1. docker image는 여러 개의 층(layer)이 쌓여서 만들어진다.
2. 각 층은 Dockerfile 명령 한 줄에 대응하고, 한 번 만들어진 층은 그대로 저장된다.
3. 층이 저장돼 있으니, 다음 빌드에서 그 명령이 안 바뀌었으면 층을 다시 만들지 않고
   저장된 것을 그대로 쓴다.
4. 다시 만들지 않는 층이 많을수록, 빌드에서 실제로 새로 하는 일이 줄어든다.
5. 새로 하는 일이 줄었으니 빌드가 빨라진다.

(2번은 1번의 "층"을, 3번은 2번의 "저장된다"를, 4번은 3번의 "다시 만들지 않는다"를,
5번은 4번의 "일이 줄어든다"를 발판으로 삼았다.)

## 하지 말 것 (체크리스트)

- [ ] 비유·은유를 썼는가? → 지운다.
- [ ] 더 쉬운 단어로 바꿀 수 있는 어려운 단어가 있는가? → 바꾼다.
- [ ] 사용자가 모른다고 가정한 실무 용어를, 먼저 설명 없이 그냥 던졌는가? → 아는
      개념으로 먼저 설명하고 연결한다.
- [ ] 한 문장 안에 근거 없이 넘어간 논리 도약이 있는가? → 문장을 쪼갠다.
- [ ] 결론을 근거보다 먼저 말했는가? → 근거부터 쌓는 순서로 고친다.

## 분량

개념 복잡도에 맞춘다. 쉬운 건 짧게, 단계가 많은 건 그만큼 단계를 다 밟는다.
쉬워 보이려고 단계를 건너뛰지 않고, 있어 보이려고 늘리지도 않는다.
