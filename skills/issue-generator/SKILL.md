# issue-generator

이번 Claude Code 세션 또는 오늘 한 작업을 요약해 **Linear에 새 이슈 한 개를 생성**하는 skill.
Notion 자동 로깅(Stop 훅)과는 완전히 별개이며, 사용자가 명시적으로 호출할 때만 동작한다.

전제: 이 환경에 Linear MCP(`claude.ai Linear` 커넥터)가 연결되어 있다.
연결이 안 되어 있으면 사용자에게 `/mcp` 로 `claude.ai Linear` 인증을 요청하고 멈춘다.

## 핵심 원칙

- **신규 이슈 생성 전용.** 기존 이슈에 코멘트/설명을 덧붙이지 않는다.
- **올리기 전 반드시 미리보기 + 승인.** 사용자가 동료들과 함께 쓰는 보드라, 잘못된 팀/내용으로
  이슈가 생기는 사고를 막는 게 최우선이다.
- **필요한 필드는 추천값과 함께 물어본다.** 사용자가 빈 값을 그대로 받아들일 수 있게 항상
  "추천: X" 형태로 기본값을 제시한다.

## 절차

### 1. 모드 판별

사용자 요청에서 둘 중 하나를 고른다.

- **세션 모드** — "지금/이번 세션 작업", "방금 한 거" 등. 현재 대화 내용이 소스다.
- **오늘 모드** — "오늘 한 거", "오늘 작업" 등. Notion 로그 DB에서 현재 프로젝트의
  오늘 기록이 소스다.

모호하면 사용자에게 어느 쪽인지 한 번 확인한다.

### 2. 작업 내용 수집

**세션 모드:**
현재 대화에서 사용자가 무엇을 요청했고 실제로 무엇을 했는지 추출한다. 실행한 도구 호출
(편집한 파일, 돌린 명령, 만든 것)을 근거로 삼고, 단순 조회(Read, ls 등)는 뺀다.
대화가 길어 앞부분이 잘렸으면 필요한 만큼 현재 세션의 transcript를 다시 읽어도 된다.

**오늘 모드:**
이 skill 디렉토리의 헬퍼를 실행해 오늘 현재 프로젝트의 Notion 로그를 가져온다.

```bash
python3 "$CLAUDE_SKILL_DIR/fetch_today.py" --project "$(basename "$PWD")"
```

- `$CLAUDE_SKILL_DIR` 가 없으면 이 SKILL.md가 있는 디렉토리 경로를 직접 쓴다
  (`~/.claude/skills/issue-generator`).
- 특정 날짜를 원하면 `--date 2026-06-10` 을 덧붙인다 (기본: 오늘).
- 출력은 JSON: `{"project", "date", "sessions": [{"title", "turns": [{"summary", "actions": [...]}]}]}`.
  `summary` 는 사람이 읽는 턴 요약, `actions` 는 그 턴에서 편집/생성한 파일·돌린 명령 등
  실제 도구 호출 목록(근거)이다. 요약할 땐 둘 다 참고하되(특히 `actions` 로 무엇을 실제로
  바꿨는지 확인), 출력은 4번 지침대로 사람이 읽기 편하게 정리한다(도구 호출 raw 나열 금지).
- `sessions` 가 비어 있으면 "오늘 이 프로젝트의 로그가 없다"는 뜻이니 사용자에게 알리고
  세션 모드로 갈지 묻는다.

### 3. 팀·프로젝트 결정 (매핑 캐시 활용)

이슈는 team이 필수다. repo↔팀 매핑 캐시를 먼저 본다.

캐시 경로: `~/.claude/issue-generator/linear_map.json`
형식: `{ "<repo폴더명>": { "teamId": "...", "teamName": "...", "projectId": "...", "projectName": "..." } }`
(`projectId`/`projectName` 은 없을 수 있음)

1. 파일을 Read 한다. 없거나 깨졌으면 `{}` 로 취급한다.
2. 키 `$(basename "$PWD")` 가 있으면 그 팀을 쓰되, 사용자에게 "팀 X에 만들게 (캐시됨), 맞아?"
   라고 **확인만** 한다.
3. 없으면 추천값과 함께 묻는다. 추천 팀은 이 순서로 고른다:
   - `list_issues` 를 `assignee: "me"` 로 호출해 사용자가 최근 만든 이슈들의 팀 중 최빈 팀
   - 그래도 모르면 `linear_map.json` 의 다른 엔트리에서 가장 자주 쓰인 팀
   - 그것도 없으면 `list_teams` 결과를 몇 개 제시
4. 사용자가 팀(필요하면 프로젝트까지) 정하면, 그 매핑을 `linear_map.json` 에 저장한다.
   디렉토리가 없으면 만든다. **이 캐시는 repo 밖(`~/.claude/`)에 두며 git에 올리지 않는다.**

### 4. 요약 생성 (title · description)

수집한 내용으로 이슈 본문을 만든다.

- **title**: 한 줄, 무엇을 한 작업인지 드러나게. 15~50자 정도.
- **description (Markdown)**:
  - 한두 문장 개요 (무엇을 왜 했는지)
  - `## 한 일` 아래 실제 변경/결과 위주 bullet. 편집한 파일·명령은 무엇을 진척시켰는지
    파악하는 근거로만 쓰고, bullet에는 파일명·명령어·코드 대신 "어떤 기능·동작이 어떻게
    바뀌었는지"를 쓴다. 조회성 작업은 제외한다.
  - 현학적인 표현을 피하고 직관적이고 쉬운 표현을 사용한다.
  - 오늘 모드면 여러 세션을 묶어 한 번 더 압축한다 (턴 나열 금지).
- 한국어로 작성. **코드·명령어·파일명 등 실무 세부는 이슈에 넣지 않는다** — 지엽적이라
  팀원이 이해하는 데 오히려 방해된다. 그 대신 팀원이 이해할 수 있는 수준의 표현으로 쓴다.
  고유명사(제품·서비스·기능명)는 원문 그대로 둔다.

### 5. 나머지 필드 질문 (추천값과 함께)

아래를 한 번에, 각 항목에 추천값을 붙여 물어본다. 사용자가 "기본값으로"라고 하면 추천값을 쓴다.

| 필드 | 추천값 |
|---|---|
| priority | None (0). 사용자가 급하다고 하면 High |
| estimate (point) | 작업량 기준 추천. 단, 팀마다 estimate 스케일/사용 여부가 다르니 모르면 "생략"을 기본으로 제시 |
| due date | 없음 (기본). cycle에 들어가면 굳이 due 안 잡아도 됨 |
| project | 3번에서 정해졌으면 그것, 아니면 없음 |
| assignee | 본인("me") |
| labels | 없음 (기본). repo명을 라벨로 붙이고 싶어 하면 제안 |

estimate 스케일이 필요하면 `get_team` 으로 팀 설정을 확인한다.

### 6. 미리보기 + 승인

생성 직전, 아래를 한 화면에 보여주고 명시적 승인을 받는다.

```
팀: <teamName>  /  프로젝트: <projectName or 없음>
제목: <title>
우선순위: <priority>  포인트: <estimate or 생략>  마감: <due or 없음>  담당: <assignee>
---
<description>
```

사용자가 고치자고 하면 반영 후 다시 보여준다. 승인 전에는 절대 생성하지 않는다.

### 7. 이슈 생성

Linear MCP `save_issue` 로 생성한다 (`id` 없이 호출 = 신규 생성).
- `team` 은 필수. `assignee` 는 `"me"` 등.
- 생성 후 반환된 이슈 **identifier(예: AI-123)와 URL을 사용자에게 출력**한다.
- 3번에서 새 매핑을 정했다면 이때 `linear_map.json` 저장이 끝났는지 확인한다.

## 주의

- 한 번 호출 = 이슈 한 개. 여러 개로 쪼개야 할 분량이면 사용자에게 어떻게 나눌지 먼저 확인한다.
- 같은 작업을 중복 등록하지 않도록, 방금 만든 이슈와 비슷한 요청이 다시 오면 한 번 짚어준다.
- Linear에 보내는 문자열은 이스케이프 시퀀스(`\n`) 대신 실제 줄바꿈을 쓴다.
