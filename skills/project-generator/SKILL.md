# project-generator

**Linear에 새 프로젝트 한 개를 생성**하는 skill. 프로젝트는 여러 이슈를 묶는 상위 단위다.
신규 이슈 생성은 `issue-generator`, 기존 이슈 코멘트는 `issue-commentor` 가 담당하고,
이 skill은 **프로젝트 생성 전용**이다. 사용자가 명시적으로 호출할 때만 동작한다.

전제: 이 환경에 Linear MCP(`claude.ai Linear` 커넥터)가 연결되어 있다.
연결이 안 되어 있으면 사용자에게 `/mcp` 로 `claude.ai Linear` 인증을 요청하고 멈춘다.

## 핵심 원칙

- **신규 프로젝트 생성 전용.** 기존 프로젝트 수정이나 이슈 신규 생성은 하지 않는다.
- **생성·이동 전 반드시 미리보기 + 승인.** 사용자가 동료들과 함께 쓰는 보드라, 잘못된
  팀/내용으로 프로젝트가 생기거나 이슈가 엉뚱한 프로젝트로 옮겨지는 사고를 막는 게 최우선이다.
- **필요한 필드는 추천값과 함께 물어본다.** 항상 "추천: X" 형태로 기본값을 제시한다.
- **이슈 이동은 별도 승인 게이트.** 프로젝트 생성과 이슈 이동은 다른 행동이므로, 모드 1에서
  이슈를 옮기기 전에 "어떤 이슈를 옮길지" 목록을 한 번 더 보여주고 승인을 받는다.

## 절차

### 1. 모드 판별

사용자 요청에서 셋 중 하나를 고른다. 모호하면 한 번 확인한다.

- **모드 1 (이슈 묶기)** — "프로젝트에 안 속한 이슈 모아서", "떠도는 이슈들 프로젝트로 묶어" 등.
  프로젝트가 없는 내 이슈들이 소스다.
- **모드 2 (세션 목적)** — "이번 세션/지금 작업 목적을 프로젝트로", "지금 하는 거 프로젝트로 만들어" 등.
  현재 Claude Code 대화 맥락이 소스다.
- **모드 3 (사용자 제시)** — "이런 프로젝트 만들어줘"처럼 사용자가 이름/목표/범위를 직접 제시.
  사용자 입력이 소스다.

### 2. 팀 결정 (매핑 캐시 활용)

프로젝트는 팀(`addTeams`)이 필수다. `issue-generator` 와 **같은 캐시**를 공유한다.

캐시 경로: `~/.claude/issue-generator/linear_map.json`
형식: `{ "<repo폴더명>": { "teamId": "...", "teamName": "...", "projectId": "...", "projectName": "..." } }`
(`projectId`/`projectName` 은 없을 수 있음)

1. 파일을 Read 한다. 없거나 깨졌으면 `{}` 로 취급한다.
2. 키 `$(basename "$PWD")` 가 있으면 그 팀을 쓰되, "팀 X에 만들게 (캐시됨), 맞아?" 라고 **확인만** 한다.
3. 없으면 추천값과 함께 묻는다. 추천 팀은 이 순서로 고른다:
   - `list_issues` 를 `assignee: "me"` 로 호출해 사용자가 최근 만든 이슈들의 팀 중 최빈 팀
   - 그래도 모르면 `linear_map.json` 의 다른 엔트리에서 가장 자주 쓰인 팀
   - 그것도 없으면 `list_teams` 결과를 몇 개 제시

### 3. 모드별 내용 수집

#### 모드 1 — 이슈 묶기

1. `list_issues` 를 `assignee: "me"`, `includeArchived: false`, `limit` 을 넉넉히(예: 100) 호출한다.
   응답에서 **`project` 가 비어 있고** 완료/취소(`Done`/`Canceled`)가 아닌 **active 이슈만** 추린다.
   - 완료된 이슈도 묶을지 사용자가 원하면 포함한다(기본은 active만).
   - 응답에 프로젝트 소속 여부가 분명치 않은 이슈는 그 이슈만 `get_issue` 로 확인한다.
2. 추린 이슈를 `identifier · 제목 · 상태` 로 보여준다. 그리고 주제를 분석한다.
   - 주제가 명확히 여러 갈래로 갈리면 **2~3개 프로젝트**로 그룹핑해 각각 이름안과 함께 제안한다.
   - 한 덩어리로 묶이면 **1개 프로젝트** 이름안을 제안한다.
   - 이름안은 항상 "추천: …" 형태로 1~2개 제시하고, 사용자가 바꿀 수 있게 한다.
3. 사용자가 **프로젝트 이름 / 팀(2번) / 포함할 이슈**를 확정한다.
   여러 프로젝트로 나누기로 했다면, 이 skill은 **한 번 호출 = 프로젝트 한 개** 원칙이므로
   먼저 한 개를 끝까지 만들고, 나머지는 이어서 만들지 사용자에게 확인한다.

#### 모드 2 — 세션 목적

1. 현재 대화에서 이번 세션이 **무엇을 이루려는 작업 묶음인지(목표·범위)** 를 추출한다.
   개별 턴 나열이 아니라, 프로젝트 단위로 묶일 만한 **상위 목적**으로 압축한다.
   대화가 길어 앞부분이 잘렸으면 필요한 만큼 현재 세션 transcript를 다시 읽어도 된다.
2. 이 목적으로 `name`/`summary`/`description` 초안을 만든다(4번 형식).

#### 모드 3 — 사용자 제시

1. 사용자가 준 이름/목표/범위를 정리한다. 빠진 정보(특히 이름)는 추천값과 함께 묻는다.
2. 정리한 내용으로 `name`/`summary`/`description` 초안을 만든다(4번 형식).

### 4. 프로젝트 본문 생성 (name · summary · description)

- **name**: 한 줄. 프로젝트가 무엇을 위한 묶음인지 드러나게.
- **summary**: 한 줄 요약(≤255자). 목록/카드에서 보일 핵심.
- **description (Markdown)**:
  - 한두 문장 개요(이 프로젝트가 무엇을, 왜).
  - `## 목표` 아래 이루려는 것 bullet.
  - 필요하면 `## 범위`(포함/제외) bullet.
  - **모드 1** 이면 `## 포함 이슈` 아래 옮길 이슈를 `identifier 제목` 으로 나열.
- 한국어로 작성. 고유명사·코드·명령어·파일명만 원문 그대로.

### 5. 나머지 필드 (추천값과 함께)

아래를 한 번에, 각 항목에 추천값을 붙여 물어본다. "기본값으로"라고 하면 추천값을 쓴다.

| 필드 | 추천값 |
|---|---|
| lead | 본인("me") |
| priority | None(0). 급하면 High |
| state | 기본 생략(Backlog/Planned 등 팀 기본). 바로 시작이면 사용자가 지정 |
| startDate / targetDate | 없음(기본). 기간이 분명하면 지정 |
| labels / initiatives | 없음(기본). 사용자가 원하면 제안 |

### 6. 미리보기 + 승인

생성 직전, 아래를 한 화면에 보여주고 명시적 승인을 받는다.

```
팀: <teamName>  /  리드: <lead>
프로젝트: <name>
요약: <summary>
상태: <state or 기본>  우선순위: <priority>  기간: <start~target or 없음>
---
<description>
(모드1) 이동할 이슈: <identifier 목록>
```

사용자가 고치자고 하면 반영 후 다시 보여준다. 승인 전에는 절대 생성하지 않는다.

### 7. 프로젝트 생성

Linear MCP `save_project` 로 생성한다 (`id` 없이 호출 = 신규 생성).
- `name` 필수. 팀은 `addTeams` 에 2번에서 정한 팀(이름 또는 ID)을 배열로 전달. **필수.**
- `lead`, `summary`, `description`, `priority` 등 5번에서 정한 값.
- 생성 후 반환된 프로젝트 **이름과 URL을 사용자에게 출력**한다.

### 8. (모드 1 전용) 이슈 이동

프로젝트 생성 직후, **별도 승인 게이트**를 둔다.

1. "이 이슈들을 `<프로젝트명>` 으로 옮길게" 목록(`identifier 제목`)을 다시 보여주고 승인받는다.
2. 승인되면 각 이슈에 대해 `save_issue` 를 `id: <identifier>`, `project: <새 프로젝트명 또는 ID>`
   로 호출해 옮긴다(`id` 가 있으므로 신규 생성이 아니라 기존 이슈 업데이트).
3. 끝나면 **옮긴 이슈 개수와 프로젝트 URL** 을 출력한다. 일부 실패하면 어떤 이슈가
   안 옮겨졌는지 알린다.

### 9. (선택) 캐시 갱신

모드 2/3 로 만든 프로젝트가 이 repo 작업과 직결되면, "이 프로젝트를 이 repo 기본
프로젝트로 캐시해 둘까? (앞으로 이슈 만들 때 추천에 씀)" 라고 묻는다. 동의하면
`~/.claude/issue-generator/linear_map.json` 의 `$(basename "$PWD")` 엔트리에
`projectId`/`projectName` 을 채운다(없으면 엔트리 생성). 이 캐시는 repo 밖(`~/.claude/`)에
두며 git에 올리지 않는다. 모드 1은 묶음용이라 캐시 갱신은 기본적으로 하지 않는다.

## 주의

- 한 번 호출 = 프로젝트 한 개. 여러 개로 나눠야 하면 한 개를 끝낸 뒤 이어갈지 확인한다.
- 같은 프로젝트를 중복 생성하지 않도록, 방금 만든 것과 비슷한 요청이 다시 오면 한 번 짚어준다.
- 이슈를 옮기기 전에는 반드시 8번의 이동 승인을 받는다. 프로젝트 생성 승인이 이슈 이동
  승인을 겸하지 않는다.
- Linear에 보내는 문자열은 이스케이프 시퀀스(`\n`) 대신 실제 줄바꿈을 쓴다.
