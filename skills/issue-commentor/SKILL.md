# issue-commentor

이번 Claude Code 세션 또는 오늘 한 작업을 요약해 **Linear의 기존 이슈에 코멘트 한 개를
등록**하는 skill. 신규 이슈를 만드는 건 `issue-generator` 가 담당하고, 이 skill은
**이미 있는 이슈에 진행 상황·결과를 덧붙이는 용도**다.
Notion 자동 로깅(Stop 훅)과는 완전히 별개이며, 사용자가 명시적으로 호출할 때만 동작한다.

전제: 이 환경에 Linear MCP(`claude.ai Linear` 커넥터)가 연결되어 있다.
연결이 안 되어 있으면 사용자에게 `/mcp` 로 `claude.ai Linear` 인증을 요청하고 멈춘다.

## 핵심 원칙

- **기존 이슈 코멘트 전용.** 새 이슈를 만들지 않는다. 코멘트 달 이슈가 불명확하면
  새로 만들지 말고 사용자에게 어느 이슈인지 묻는다.
- **올리기 전 반드시 "대상 이슈 + 코멘트 본문" 미리보기 + 승인.** 사용자가 동료들과 함께
  쓰는 보드라, 엉뚱한 이슈에 코멘트가 달리는 사고를 막는 게 최우선이다.
- **대상 이슈를 추측으로 확정하지 않는다.** identifier(예: AI-123)나 URL이 명확하지
  않으면, 후보를 제시하고 사용자가 고르게 한다.

## 절차

### 1. 대상 이슈 특정 (이 skill의 핵심)

코멘트를 어디에 달지부터 정한다. 아래 순서로 시도한다.

1. **사용자가 identifier나 URL을 직접 준 경우** (예: "AI-123에", "이 링크 이슈에"):
   `get_issue` 로 그 이슈를 조회해 제목·상태·담당자를 확보한다.
2. **"방금/아까 만든 이슈", "그 이슈" 처럼 이번 세션 맥락을 가리키는 경우**:
   현재 대화에서 직전에 생성·언급된 이슈 identifier를 찾아 `get_issue` 로 확인한다.
   대화가 길어 잘렸으면 필요한 만큼 세션 transcript를 다시 읽어도 된다.
3. **대상이 모호한 경우**:
   `list_issues` 를 `assignee: "me"` 로 호출해(필요하면 `query` 로 키워드 좁히기)
   최근 이슈 몇 개를 `identifier · 제목 · 상태` 형태로 제시하고 사용자가 고르게 한다.

확정 전 항상 **"이 이슈에 달게: <identifier> <제목> (상태: <state>)"** 로 한 번 확인한다.
대상이 끝까지 불명확하면 멈추고 묻는다. 임의로 새 이슈를 만들지 않는다.

### 2. 작업 내용 수집

`issue-generator` 와 동일한 두 모드를 쓴다. 요청에서 하나를 고른다.

- **세션 모드** — "지금/이번 세션 작업", "방금 한 거" 등. 현재 대화 내용이 소스다.
  사용자가 무엇을 요청했고 실제로 무엇을 했는지(편집한 파일, 돌린 명령, 만든 것)를
  근거로 추출하고, 단순 조회(Read, ls 등)는 뺀다.
- **오늘 모드** — "오늘 한 거", "오늘 작업" 등. Notion 로그 DB에서 현재 프로젝트의
  오늘 기록이 소스다. 이 skill 디렉토리의 헬퍼를 실행해 가져온다.

```bash
python3 ~/.claude/skills/issue-commentor/fetch_today.py --project "$(basename "$PWD")"
```

- `$CLAUDE_SKILL_DIR` 가 있으면 `"$CLAUDE_SKILL_DIR/fetch_today.py"` 를 써도 된다.
- 이 `fetch_today.py` 는 repo 루트의 공통 모듈 `notion_today.py` 를 호출하는 얇은
  진입점이다. issue-generator 와 로직을 공유하지만 그쪽 경로에 의존하지는 않는다.
- 특정 날짜를 원하면 `--date 2026-06-10` 을 덧붙인다 (기본: 오늘).
- 출력은 JSON: `{"project", "date", "sessions": [{"title", "turns": [{"summary", "actions": [...]}]}]}`.
  `summary` 는 사람이 읽는 턴 요약, `actions` 는 그 턴에서 편집/생성한 파일·돌린 명령 등
  실제 도구 호출 목록(근거)이다. 코멘트를 쓸 땐 둘 다 참고하되(특히 `actions` 로 무엇을
  실제로 진척시켰는지 확인), 출력은 3번 지침대로 사람이 읽기 편하게 정리한다(도구 호출 raw 나열 금지).
- `sessions` 가 비어 있으면 "오늘 이 프로젝트의 로그가 없다"는 뜻이니 사용자에게 알리고
  세션 모드로 갈지 묻는다.

### 3. 코멘트 본문 생성 (Markdown)

수집한 내용으로 코멘트 본문을 만든다. 코멘트는 이슈에 **덧붙는 진행 메모**이므로
제목 필드가 없고, 이슈를 처음부터 설명할 필요 없이 "이번에 무엇을 진척시켰는지"에 집중한다.

- 맨 앞에 한 줄 요약 (이번에 한 일의 핵심).
- 그 아래 `## 한 일` bullet — 실제 변경/결과 위주. 편집한 파일·명령은 무엇을 진척시켰는지
  파악하는 근거로만 쓰고, bullet에는 파일명·명령어·코드 대신 "어떤 기능·동작이 어떻게
  바뀌었는지"를 쓴다. 조회성 작업은 제외한다.
- 현학적인 표현을 피하고 직관적이고 쉬운 표현을 사용한다.
- 후속 작업이나 막힌 지점이 있으면 `## 다음/이슈` 로 짧게 덧붙인다(없으면 생략).
- 오늘 모드면 여러 세션을 묶어 한 번 더 압축한다(턴 나열 금지).
- 한국어로 작성. **코드·명령어·파일명 등 실무 세부는 이슈에 넣지 않는다** — 지엽적이라
  팀원이 이해하는 데 오히려 방해된다. 그 대신 팀원이 이해할 수 있는 수준의 표현으로 쓴다.
  고유명사(제품·서비스·기능명)는 원문 그대로 둔다.

### 4. 미리보기 + 승인

등록 직전, 아래를 한 화면에 보여주고 명시적 승인을 받는다.

```
대상 이슈: <identifier> <제목>  (상태: <state>, 담당: <assignee>)
---
<comment body>
```

사용자가 고치자고 하면 반영 후 다시 보여준다. 승인 전에는 절대 등록하지 않는다.

### 5. 코멘트 등록

Linear MCP `save_comment` 로 등록한다 (`id` 없이 호출 = 신규 코멘트).
- `issueId` 는 1번에서 확정한 identifier(예: AI-123)나 이슈 ID. **필수.**
- `body` 는 3번에서 만든 Markdown.
- 등록 후, 대상 이슈의 **identifier와 URL을 사용자에게 출력**한다.

## 주의

- 한 번 호출 = 코멘트 한 개. 한 이슈에 같은 작업을 중복으로 코멘트하지 않도록,
  방금 단 코멘트와 비슷한 요청이 다시 오면 한 번 짚어준다.
- 대상 이슈가 불명확하거나 "새 이슈가 맞겠다" 싶으면, 코멘트를 강행하지 말고
  `issue-generator` 사용을 사용자에게 제안한다.
- Linear에 보내는 문자열은 이스케이프 시퀀스(`\n`) 대신 실제 줄바꿈을 쓴다.
